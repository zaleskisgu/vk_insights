# План разработки (по [tz.md](./tz.md) и текущему коду)

Опираемся на [IMPLEMENTATION.md](./IMPLEMENTATION.md): что уже есть, что закрывать дальше.

## Приоритет 1 (ядро сдачи)

1. **Реальный `HttpVkClient`**
   - `groups.getById`, `wall.get` с `v`, `access_token` из `config/vk.php`.
   - **Пагинация** `wall.get` (offset/while `count` == 100) до исчерпания **или** до границы периода.
   - Фильтр постов по **датам** `from` / `to` (unix) после выборки или в рамках запросов, как удобнее с лимитами.
   - Ошибки VK: `error` в JSON, **429** / сеть / таймауты — осмысленные HTTP-ответы и (при 429) retry/backoff по политике.
   - В `AppServiceProvider` — переключение: **`MockVkClient` | `HttpVkClient`** от `config('vk.use_mock')` (и проверка `VK_SERVICE_TOKEN` при `use_mock = false`).

2. **Контракт и данные отчёта**
   - Вход: ** `group_id` / screen name / ссылка** (парсер → числовой id и/или `owner_id` = `-id`).
   - Период: **`from` / `to`**.
   - Выход агрегатов: топ по вовлечённости (likes + comments + reposts), **средняя вовлечённость за период**, **тип поста** (текст/фото/видео/ссылка и пр.) по правилам из `attachments` и `post_type`, **динамика по дням** (сумма вовлечения или кол-во постов в день — зафиксировать в API).

3. **Кэш** (10–30 мин)
   - Ключ: например `hash(groupId, from, to)`; драйвер — `file` / `database` / `redis` по [tz.md](./tz.md); TTL из конфига.
   - Промах: VK → расчёт → **запись в кэш**.

4. **Согласовать REST с ТЗ**
   - `POST /analyze` — старт/обновление (тяжёлая часть + кэш).
   - `GET /report/{groupId}?from=&to=` — **чтение** готового отчёта (сначала кэш) или 404/подсказка «сначала analyze».
   - Либо **обосновать** один эндпоинт в README, если оставляете иной дизайн.
   - **`GET /health`**: маршрут-алиас или `Route::get('/health', ...)` рядом с встроенным `GET /up` — чтобы совпадало с [tz.md](./tz.md).

5. **Наблюдаемость**
   - Лог: время обработки запроса, **число HTTP-вызовов** к `api.vk.com`, (опц.) `execute` batch.

## Приоритет 2 (интеграция с фронтом)

6. **Формат отчёта для дашборда** — стабильная JSON-схема: мета периода, срез по дням, таблица топов, сводка по типам, флаги ошибок VK.
7. **Экспорт** — `GET` или `POST` с `Accept: text/csv` / `application/json` (или query `format=csv|json` с учётом кэша и `GET`/`POST`) по [tz.md](./tz.md).

## Приоритет 3 (фронт + оптимизация)

8. **Фронт** — ввод сообщества + период; таблица + **минимум один график**; **loading / empty / error**; адаптив. Стек на выбор (React/Vue/Svelte — [tz.md](./tz.md)).
9. **PERF.md** — 2–3 пункта «**проблема → решение → эффект**» + **Lighthouse** или **Web Vitals** до/после.
10. **README** — запуск, `VK_SERVICE_TOKEN`, `API_TOKEN`, мок/прод, ссылка на демо/GIF (опц.).

## Соответствие критериям (напоминание из [tz.md](./tz.md))

| Критерий     | Как закрывать плану |
| ------------ | -------------------- |
| VK 35%       | Пагинация, 429, стабильность, мок для разработки |
| Backend 25%  | Кэш, явные маршруты, чистая связка `VkClient` + сервис отчёта |
| UX 20%       | Состояния, читаемый дашборд |
| Оптимиза 20% | `PERF.md` с цифрами |

*Если сроки жмут: ТЗ прямо допускает урезать объём, приоритезируя **VK + кэш + базовый дашборд**.*
