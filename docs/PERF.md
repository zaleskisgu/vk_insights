# Оптимизация фронтенда (измеримо)

Формат: **проблема → решение → измеримый эффект**. Замеры после `npm run build` (Vite 8), клиентская среда production, май 2026.

## 1. Тяжёлый дашборд в начальном бандле

**Проблема.** Пользователь на первом экране вводит сообщество и период; код графиков (Chart.js), таблиц PrimeVue и всех виджетов дашборда не нужен до успешного запроса отчёта (`GET /report`). Раньше весь этот код шёл в один entry-чанк вместе с приложением.

**Решение.** Экран [`DashboardScreen.vue`](../resources/js/screens/DashboardScreen.vue) подключается в [`App.vue`](../resources/js/App.vue) через `defineAsyncComponent` + `<Suspense>` с коротким fallback «Загрузка отчёта…». Vite выносит дашборд в отдельный async-чанк (см. `dynamicImports` в `public/build/manifest.json`).

**Измеримый эффект.**

| Чанк (тип в `manifest.json`) | Размер (gzip) |
| --------------------------- | ------------- |
| Entry `app-*.js` | ~428 kB (~98 kB gzip) |
| Async `DashboardScreen-*.js` | ~436 kB (~97 kB gzip) |
| Динамический Chart.js `auto-*.js` | ~203 kB (~70 kB gzip) |

Имена файлов с хешем (`app-hCAoi3Bs.js` и т.д.) меняются после каждой сборки; порядок величин и разбиение на чанки сохраняются.

Первая отрисовка стартового экрана не парсит и не исполняет код дашборда до запроса отчёта: сеть тянет только entry + CSS + иконки; после появления отчёта подгружается отдельный JS дашборда (и при необходимости Chart.js). Это снижает объём критического пути для «холодного» захода на главную.

## 2. Импорты PrimeVue по компонентам

**Проблема.** Подключение всей библиотеки целиком раздувает бандл и ухудшает время до интерактивности.

**Решение.** Во всех Vue-компонентах используются точечные импорты вида `import Button from 'primevue/button'`, а не монолитный пакет.

**Измеримый эффект.** Tree-shaking Vite оставляет в entry только реально используемые модули PrimeVue; точный выигрыш от «полного импорта» здесь не мерился (контрольной ветки в репозитории нет), но подход зафиксирован как устойчивая практика для роста UI.

## 3. Общая тема Chart.js

**Проблема.** Дублирование настроек цветов осей и легенды в трёх компонентах графиков усложняло сопровождение и могло привести к рассинхрону стилей.

**Решение.** Вынесены константы в [`resources/js/utils/chartTheme.js`](../resources/js/utils/chartTheme.js); компоненты графиков импортируют общие фрагменты опций.

**Измеримый эффект.** Прямой выигрыш по килобайтам минимален (несколько сотен байт исходника); выигрыш — предсказуемость и меньше дублирования при добавлении новых графиков (меньше риска лишних diff в бандле из-за расхождений).

---

**Как воспроизвести замеры:** из корня репозитория `npm run build`, затем смотреть таблицу gzip в выводе Vite и файл [`public/build/manifest.json`](../public/build/manifest.json).
